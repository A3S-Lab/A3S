# Box And Tools

`a3s box ...` is the umbrella CLI's Box proxy. It keeps the common entry point
simple while leaving sandbox/runtime behavior in the `a3s-box` binary.

## Box Proxy Flow

When you run `a3s box <args...>`, the CLI:

1. Looks for an executable `a3s-box`.
2. Checks `A3S_BOX_INSTALL_DIR` first when it is set.
3. Checks next to the current `a3s` executable.
4. Checks `PATH`.
5. Checks a user-local fallback location.
6. Installs Box when missing, then re-runs the requested command.

The proxy exits with the same status code as `a3s-box` when the child command
fails.

## Bootstrap Paths

| Path | Behavior |
| --- | --- |
| Homebrew | Tries the `a3s-lab/tap/a3s-box` formula when `brew` is available. |
| Standalone release | Downloads a platform release archive, extracts Box binaries, installs them into the chosen bin directory, and copies bundled libraries. |
| Existing install | Reuses an existing executable from the configured install dir, CLI sibling dir, `PATH`, or user-local fallback. |

Standalone Box bootstrap currently has release targets for macOS arm64, Linux
arm64, and Linux x86_64. Unsupported OS/arch pairs should install Box manually
from the release page.

## Installed Box Binaries

When present in the release archive, bootstrap installs:

- `a3s-box`
- `a3s-box-shim`
- `a3s-box-guest-init`
- `a3s-box-cri`

## Component And Tool Listing

`a3s list` reports the typed catalog and registered external tools without
installing, upgrading, or executing them. The catalog distinguishes bundled
Code, first-use Box and Use, explicitly installed Bench and Search, and
delegated Use capabilities.

## A3S Use First-Use Flow

Installing the umbrella CLI does not install A3S Use. The first
`a3s use ...` command or `a3s code` TUI startup resolves a healthy Use receipt
or downloads and verifies the complete platform release when policy allows.
The release keeps these assets together:

- `a3s-use` and its Browser driver;
- Browser, Document, Office, and OCR Skills;
- the PP-OCRv6 Small detection and recognition ONNX models; and
- the Browser Dashboard and license/provenance files.

Subsequent capability preparation is idempotent. Browser shares the same
`a3s-use-browser` Chrome directory with Search. Native Office needs no provider;
only optional compatibility commands prepare OfficeCLI. Document and OCR share
one PP-OCRv6 model directory, so preparing either does not create a duplicate
model installation.

```bash
a3s install use/browser   # optional Browser pre-warm
a3s install use/document  # optional shared PP-OCRv6 pre-warm
a3s install use/office    # optional OfficeCLI compatibility
a3s install use/ocr       # same PP-OCRv6 bundle as Document
```

`--offline`, `A3S_OFFLINE=1`, and `A3S_NO_AUTO_INSTALL=1` disable first-use
downloads. Explicit `a3s install ...` commands remain available for machines
that must be prepared before going offline.

## Boundaries

The CLI does not implement container isolation or application automation
itself. It discovers, installs, and launches Box and Use; their repositories
own runtime behavior, Browser automation, native document parsing, and local
OCR.
