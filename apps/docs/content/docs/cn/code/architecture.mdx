# 架构

A3S Code 围绕小运行时内核和控制执行循环的驾驭层组织。

```text
Prompt
  -> context assembly
  -> optional planning and automatic subagent delegation
  -> LLM generation
  -> model-selected、delegated 或 programmatic tool calls
  -> permission、hook、AHP、confirmation 与 security checks
  -> tool execution
  -> observation
  -> verification
  -> trace storage
  -> compaction
```

内核负责维护 session、调用 provider、通过同一个 executor 执行工具、记录 observation，并返回结构化事件和结果。工具调用可以来自模型、委派子运行，或 `program` 工具。SDK 直接 helper 是另一条宿主控制面入口：宿主先授权，再调用 SDK；随后它们使用同一个 registry、workspace services、artifact 和结果形状。

驾驭层协调上下文预算、工具可见性、permission/hooks/AHP/security-provider 集成点、输出摘要、完成验证，以及面向宿主应用暴露 run snapshot、TaskList 状态和 trace event。

这个边界很重要：runtime 不会把 TUI 当成特殊路径。`a3s code` 只是把这些事件流渲染到终端的一个客户端；SDK 宿主可以在自己的产品界面中渲染、审批、持久化和回放同一组事实。

## PTC 路径

Programmatic Tool Calling 是确定性工作流的驾驭路径。`program` 工具会在内嵌 QuickJS VM 中运行 JavaScript 脚本，并只暴露一个受控 `ctx` 对象。当模型在 agent turn 内调用 `program` 时，它和其他模型选择的工具一样经过 permission、hook/AHP、confirmation、timeout、artifact、trace 和 workspace-capability 路径。宿主代码直接调用 `session.program(...)` 时，它是 direct SDK call，必须先由宿主授权。默认情况下，脚本可以调用除 `program` 之外的所有已注册工具；调用方也可以用 `allowed_tools` 收窄工具面。脚本可以组合 `readFile`、`read`、`glob`、`grep`、`ls`、`bash`、`git` 和通用 `tool(name, args)`，并把一个结构化结果交给 agent。

适合使用 PTC 的场景包括：

- 仓库搜索与代码片段收集
- 测试发现与验证命令选择
- 包元数据检查
- 发布 preflight 证据收集
- 日志解析与摘要
- trace 索引

当下一步需要判断时，让模型调用工具；当动作序列已经确定时，用 PTC 执行，再让模型消费结果。脚本可以通过内联 `source` 传入，也可以通过 workspace 相对 `.js` 或 `.mjs` 文件路径载入；VM 本身没有直接文件系统、网络、子进程或环境变量访问权限。
