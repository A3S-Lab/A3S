# A3S Gateway

A3S Gateway 是 AI 原生服务的流量层。它基于 [hyper](https://hyper.rs/) 和 [rustls](https://github.com/rustls/rustls) 构建，以单一静态链接二进制发布，配置完全使用 ACL（Agent Configuration Language）定义。配置文件必须使用 `.acl` 扩展名 —— HCL、TOML、YAML 和 JSON 配置文件在加载时会被拒绝。

本页是一份概览。下文每项能力都链接到完整记录它的页面：

- [快速开始](./quick-start) — 安装并运行你的第一个网关
- [Wire Firewall](./wire-firewall) — 可选 inline LLM/MCP proxy，在转发前 mask secrets 并运行 Sentry inspection
- [配置](./configuration) — 完整的 ACL schema（entrypoints、routers、services、middlewares、providers、management、observability）
- [路由](./routing) — Traefik 风格的规则匹配与优先级
- [中间件](./middleware) — 内置的请求/响应管道
- [服务](./services) — 负载均衡、健康检查、会话保持、故障转移和 AI 原生扩缩容
- [可观测性](./observability) — Prometheus 指标、访问日志、追踪和管理 API
- [部署](./deployment) — Docker、Helm/Kubernetes、Homebrew、Cargo

## 为什么选择 A3S Gateway

AI 服务与 Web 应用有根本不同的运维模式。为 Web 时代设计的工具在应用于 AI 技术栈时存在先天假设偏差：

| 关注点 | Web 的假设 | AI 的现实 |
|-------|-----------|---------|
| 响应形状 | 小而有界的 JSON | 无界 Token 流（SSE） |
| 延迟特征 | 毫秒级 | 秒级（模型推理） |
| 空闲成本 | 可忽略 | 高（GPU 显存占用） |
| 部署风险 | 低，无状态 | 高（模型质量回退） |
| 协议需求 | HTTP 请求/响应 | SSE、WebSocket、gRPC、HTTP/2 |

A3S Gateway 为右列而设计：

- **SSE/流式传输** — 零响应缓冲的分块传输中继。模型产出第一个 Token 时，客户端立刻收到，而不是等待完整响应组装完毕。SSE 通过 `Accept: text/event-stream` 检测（也包括 `application/x-ndjson`、`application/stream+json` 以及分块 `Transfer-Encoding`）。
- **缩容到零 + 冷启动缓冲** — 服务可以将副本下限缩容到零（`min_replicas = 0`）。启用缓冲后（`buffer_enabled = true`），到达时没有健康后端的请求会被*保持*在一个有界的内存队列中，并在后端就绪的瞬间重新派发 —— 是把进行中的请求保持住，而不是先记录再回放。
- **版本流量分割** — 按权重在命名版本之间分割实时流量（`traffic_percent` 之和必须为 100）。选择采用确定性加权轮询，不健康的版本会被跳过，回退到下一个拥有健康后端的版本。
- **流量镜像** — 将可配置 `percentage` 的实时请求复制到影子服务，发出即不管。影子响应会被丢弃，绝不影响客户端的回复或延迟。
- **熔断器** — 当 AI 后端开始返回 5xx（推理变慢、GPU OOM）时，自动开启熔断，停止放大问题。

其余部分——路由、TLS、限流、认证、Prometheus——是标准基础设施，A3S Gateway 已内置，无需引入第二个工具。

<Callout type="info">
**灰度发布**（`rollout { }`）已完整实现并通过验证 —— `from`/`to` 版本、`step_percent`、`step_interval_secs`，以及在 `error_rate_threshold` / `latency_threshold_ms` 触发时的自动回滚。在 v1.0.11 中，灰度发布控制器**已被解析和验证，但尚未由运行时循环驱动**，因此该配置在运行时是惰性的。当前请使用静态[版本分割](./services)手动切换流量。
</Callout>

## 架构

```
Client Request
    ↓
Entrypoint  (HTTP / HTTPS / TCP / UDP listener)
    ↓
TLS Termination  (rustls, static cert/key; min TLS 1.2 or 1.3)
    ↓
Router  (Host / Path / PathPrefix / Headers / Method matching, && only)
    ↓
Middleware Pipeline  (request order in, reverse order out; short-circuit allowed)
    ↓
Service  (load balancer + active/passive health + sticky + failover + mirror)
    ↓
Scaling  (Knative autoscaler · cold-start request buffer · revision router)
    ↓
Proxy  (HTTP / WebSocket / SSE / gRPC h2c / TCP / UDP)
    ↓
Backend Service
```

热重载通过共享运行时快照原子替换路由表和服务注册表。当 listener 配置不变（且不存在 UDP entrypoint）时，HTTP/TCP socket *不会*重新绑定 —— 运行时状态会就地热替换。新增或变更的 HTTP/TCP listener 按名称协调，未变化的 listener 即便新的绑定失败也会保持活跃。任何 UDP entrypoint 都会强制完全重启。

## 网关生命周期

```rust
pub enum GatewayState {
    Created,     // Configured, listeners not yet bound
    Starting,    // Binding ports, compiling routes and middleware pipelines
    Running,     // Accepting and proxying requests
    Reloading,   // Atomically swapping router table and service registry
    Stopping,    // Draining in-flight requests
    Stopped,     // All listeners closed
}
```

## 主要特性

**AI 原生模式** — 参见[服务](./services)
- SSE/流式中继 — 针对 LLM 输出的零缓冲分块传输
- Inline wire firewall — feature-gated `/wire/<agent>/...` proxy，用于 secret/PII masking 和 Sentry request inspection
- 缩容到零 — Knative 自动扩缩容公式，可选冷启动请求缓冲（`buffer_enabled`，默认关闭）
- 版本分割 — 在命名版本之间进行确定性加权流量分割（之和必须为 100%）
- 流量镜像 — 将一定比例的实时流量复制到影子服务，发出即不管
- 灰度发布 — 逐步流量切换，错误率或延迟触发时自动回滚（已实现并通过验证；v1.0.11 中尚未在运行时驱动）

**核心代理** — 参见[路由](./routing)和[服务](./services)
- 兼容 Traefik 的规则语法 — `Host()`、`Path()`、`PathPrefix()`、`Headers()`、`Method()`，以 `&&` 连接（仅 AND）
- 路由器优先级 — 更高的正 `priority` 胜出（Traefik 风格）；否则最长的规则字符串胜出
- 4 种负载均衡策略 — 轮询（默认）、加权、最少连接、随机
- 主动 HTTP 健康探测 + 始终开启的被动 5xx 剔除，并在后台进行半开恢复
- 会话保持 — 基于 Cookie 的亲和（Cookie 名称可配置；TTL 1h 且容量固定）
- 故障转移 — 主服务无任何健康后端时自动切换到备用服务
- 转发头 — 上游 HTTP 请求无条件添加 `X-Forwarded-For`、`-Host`、`-Proto`、`-Port`
- 纯 Rust TLS（rustls）— 静态 cert/key PEM，ALPN `h2` + `http/1.1`，最低 TLS 1.2 或 1.3
- 热重载 — `inotify`/`kqueue` 文件监听；HTTP/TCP entrypoint 不变时不重绑流量端口

**中间件（14 种 HTTP 类型 + TCP 过滤器）** — 参见[中间件](./middleware)
- 认证：API Key、Basic Auth、JWT（HS256）、Forward Auth
- 流量：限流（进程内）、限流（Redis，`redis` 特性）、熔断器、重试、请求体大小限制
- 转换：CORS、Headers、Strip Prefix（单段 `/*` 通配符）、压缩（br/gzip/deflate，仅标记）
- 网络：IP 白名单，外加在 entrypoint 上配置的连接级 TCP 过滤器（`max_connections`、`tcp_allowed_ips`）

**可观测性** — 参见[可观测性](./observability)
- Prometheus 指标 — 每路由器/服务/后端的请求计数、响应状态分类、字节数和活跃连接数（进程内，无依赖）
- 分布式追踪 — 入站提取 W3C Trace Context 和 B3 / Zipkin；出站注入 W3C `traceparent`
- 管理/Dashboard API — 可选的专用监听器（默认禁用，绑定回环地址），含 bearer token、IP 白名单和可选 TLS/mTLS

**运维** — 参见[部署](./deployment)
- 单一静态链接二进制 — 无运行时依赖，无 OpenSSL
- 仅 ACL 配置 — 人类可读、可版本控制（必须使用 `.acl` 扩展名）
- 通过文件监听热重载；可选 file/discovery/kubernetes/docker 配置 provider
- 面向 Kubernetes 的 Helm Chart、Docker 镜像（`ghcr.io/a3s-lab/gateway`）、Homebrew 公式、`cargo install a3s-gateway`
- 当前版本：**v1.0.11**（MSRV Rust 1.88）；发布的 Linux 镜像/二进制启用 `kube` + `redis` 特性
