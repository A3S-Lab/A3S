# Compress

`compress` 在响应方向把符合条件的响应标记为可压缩。实际压缩由代理层完成，并根据客户端 `Accept-Encoding` 选择 `br`、`gzip`、`deflate` 或 `identity`。

## 适用场景

- API 返回较大的 JSON、文本、HTML、CSS、JavaScript、SVG 或 XML。
- 前端资源或文档接口需要减少传输体积。
- 希望在网关层统一启用压缩，而不是让每个上游服务单独处理。

## 工作方式

请求方向直通。响应方向如果尚未设置 `Content-Encoding`，中间件会插入：

```text
x-gateway-compress: eligible
```

代理层读取该头后进行压缩协商。编码优先级为：

```text
br > gzip > deflate > identity
```

## 配置

```text
middlewares "compress" {
  type = "compress"
}
```

| 键 | 类型 | 默认值 | 说明 |
|-----|------|---------|-------|
| `type` | string | _必填_ | 固定为 `compress`。 |

## 当前限制

`compress` 在 ACL 中会忽略所有额外配置。源码中的默认最小尺寸是 `1024` 字节，压缩级别是 `6`，但当前没有 ACL 键可修改它们。

如果响应已经带有 `Content-Encoding`，中间件不会再标记压缩，避免重复压缩。

## 示例

```text
middlewares "api-compress" {
  type = "compress"
}

routers "docs-api" {
  rule        = "Host(`docs.example.com`)"
  service     = "docs-api"
  middlewares = ["cors", "api-compress"]
}
```

## 注意事项

- 小响应压缩收益有限，可能增加 CPU 成本。
- 图片、视频、zip、wasm 等已经压缩的内容通常不适合再次压缩。
- 如果上游已经做了压缩并设置 `Content-Encoding`，网关不会重复标记。
- 对延迟敏感接口启用压缩前，建议观察 CPU 和 P95/P99 延迟。

## 推荐顺序

压缩一般放在响应链路最后一类转换中：

```text
middlewares = ["cors", "auth-jwt", "headers", "compress"]
```
