# Forward Auth

`forward-auth` 把认证决策委托给外部 HTTP 服务。网关在转发业务请求前，先向认证服务发起一次 `GET` 校验；认证服务返回 `2xx` 则放行，返回非 `2xx` 则把认证服务响应原样返回给客户端。

## 适用场景

- 需要接入 OIDC、OAuth2、SSO、RBAC 或企业 IAM。
- 网关只负责流量入口，复杂认证逻辑放在独立认证服务中。
- 需要由认证服务向上游传递 `X-User-Id`、`X-User-Role`、`X-Tenant-Id` 等上下文。

## 工作方式

网关向 `forward_auth_url` 发起 `GET` 请求，并复制原始请求头，同时附加：

- `X-Forwarded-Method`：原请求方法。
- `X-Forwarded-Uri`：原请求 URI。

认证服务返回 `2xx` 时，网关继续请求；若 `forward_auth_response_headers` 中列出的响应头存在，会复制到上游请求。返回非 `2xx` 时，网关使用认证服务的状态码和响应体短路。认证服务不可达时返回 `502`：

```json
{"error":"Auth service unavailable"}
```

## 配置

```text
middlewares "ext-auth" {
  type                          = "forward-auth"
  forward_auth_url              = "https://auth.example.com/verify"
  forward_auth_response_headers = ["X-User-Id", "X-User-Role"]
}
```

| 键 | 类型 | 默认值 | 说明 |
|-----|------|---------|-------|
| `type` | string | _必填_ | 固定为 `forward-auth`。 |
| `forward_auth_url` | string | _必填_ | 外部认证端点；为空会导致配置加载失败。 |
| `forward_auth_response_headers` | array&lt;string&gt; | `[]` | 从 `2xx` 认证响应复制到上游请求的头名，复制时会转为小写头名。 |

## 示例

```text
middlewares "sso-auth" {
  type                          = "forward-auth"
  forward_auth_url              = "https://auth.example.com/verify"
  forward_auth_response_headers = ["X-User-Id", "X-User-Email", "X-Tenant-Id"]
}

routers "workspace" {
  rule        = "Host(`workspace.example.com`)"
  service     = "workspace"
  middlewares = ["sso-auth", "rate-limit", "compress"]
}
```

## 认证服务契约

认证服务应尽量保持响应简单稳定：

- `2xx`：认证通过，可选返回用户上下文头。
- `401`：未登录或令牌无效。
- `403`：已登录但无权限。
- `5xx`：认证系统异常。网关不会失败放行，而是把非 `2xx` 返回给客户端。

## 注意事项

- `forward-auth` 会在每个请求上增加一次外部 HTTP 调用，建议认证服务靠近网关部署，并设置合理超时和扩缩容。
- 认证服务不可达会返回 `502`，不是放行。
- 只复制 `forward_auth_response_headers` 列出的头，避免把认证服务内部头泄露给上游。
- 如需对公开静态资源跳过认证，建议用独立路由拆分，而不是在认证服务里做大量路径例外。

## 推荐顺序

如果认证服务需要看到原始路径，通常把 `forward-auth` 放在 `strip-prefix` 前：

```text
middlewares = ["ip-allow", "sso-auth", "rate-limit", "strip-prefix"]
```
