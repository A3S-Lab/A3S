# Circuit Breaker

`circuit-breaker` 根据后端响应状态维护熔断状态。当后端持续返回 `5xx` 时，网关会暂时拒绝新的请求，避免失败扩散和资源继续被打满。

## 适用场景

- 后端服务偶发雪崩，需要快速止损。
- 多个上游共享依赖，某个依赖异常时需要保护网关和调用方。
- 与重试、被动健康检查一起形成韧性访问策略。

## 工作方式

熔断器有三种状态：

- `Closed`：正常放行请求。
- `Open`：拒绝请求，返回 `503`。
- `HalfOpen`：冷却时间结束后允许探测请求，根据结果恢复或重新打开。

响应方向会记录后端结果：状态码 `>= 500` 计为失败，`2xx/3xx/4xx` 计为成功。连续失败达到阈值后进入 `Open`；冷却时间结束后转入 `HalfOpen`；半开状态下连续成功达到阈值后恢复 `Closed`。

`Open` 时返回：

```json
{"error":"Service unavailable (circuit breaker open)"}
```

HTTP 状态码为 `503`。

## 配置

```text
middlewares "circuit-breaker" {
  type              = "circuit-breaker"
  failure_threshold = 5
  cooldown_secs     = 30
  success_threshold = 1
}
```

| 键 | 类型 | 默认值 | 说明 |
|-----|------|---------|-------|
| `type` | string | _必填_ | 固定为 `circuit-breaker`。 |
| `failure_threshold` | u32 | `5` | 连续多少次 `5xx` 后打开熔断。 |
| `cooldown_secs` | u64 | `30` | Open 状态保持多久后进入 HalfOpen。 |
| `success_threshold` | u32 | `1` | HalfOpen 状态需要多少次成功才关闭熔断。 |

## 示例

```text
middlewares "orders-breaker" {
  type              = "circuit-breaker"
  failure_threshold = 3
  cooldown_secs     = 20
  success_threshold = 2
}

routers "orders" {
  rule        = "Host(`api.example.com`) && PathPrefix(`/orders`)"
  service     = "orders"
  middlewares = ["auth-jwt", "orders-breaker", "retry"]
}
```

## 注意事项

- `4xx` 被视为成功，因为它通常代表客户端请求问题，而不是后端不可用。
- 熔断状态是网关进程内状态；多副本之间不共享。
- 熔断器记录的是挂载该中间件实例的后端响应。不要把不相关后端共用同一个熔断中间件名称。
- 与 [Retry](/cn/docs/gateway/middleware/retry) 组合时，要避免重试把故障后端打得更重；建议重试次数小、熔断阈值敏感。

## 推荐顺序

常见顺序：

```text
middlewares = ["auth-jwt", "rate-limit", "circuit-breaker", "retry"]
```
