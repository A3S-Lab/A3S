# Retry

`retry` 中间件本身不直接发起重试。它把重试策略写入内部请求头，由代理层在转发到后端时读取并执行。

## 适用场景

- 后端偶发网络抖动或短暂 `5xx`，重试能提高成功率。
- 与服务发现、负载均衡、被动健康检查一起提升请求韧性。
- 对幂等请求增加有限重试。

## 工作方式

请求方向注入两个内部头：

```text
x-gateway-retry-max: <max_retries>
x-gateway-retry-interval-ms: <retry_interval_ms>
```

中间件不短路请求，也不会在自身逻辑里判断状态码。真正重试由代理层根据这些头完成。

## 配置

```text
middlewares "retry" {
  type              = "retry"
  max_retries       = 3
  retry_interval_ms = 100
}
```

| 键 | 类型 | 默认值 | 说明 |
|-----|------|---------|-------|
| `type` | string | _必填_ | 固定为 `retry`。 |
| `max_retries` | u32 | `3` | 最大重试次数，不包含首次请求；不能为 `0`。 |
| `retry_interval_ms` | u64 | `100` | 重试间隔，单位毫秒。 |

## 示例

```text
middlewares "safe-retry" {
  type              = "retry"
  max_retries       = 2
  retry_interval_ms = 150
}

routers "read-api" {
  rule        = "Host(`api.example.com`) && PathPrefix(`/read`)"
  service     = "read-api"
  middlewares = ["auth-jwt", "rate-limit", "safe-retry"]
}
```

## 注意事项

- 只建议对 GET、查询类 POST 或业务明确幂等的接口启用重试。
- 非幂等写入请求可能因为重试造成重复提交。
- `max_retries = 0` 会导致配置加载失败。
- 重试会增加后端压力；若后端已经大量失败，搭配 [Circuit Breaker](/cn/docs/gateway/middleware/circuit-breaker) 更稳。

## 推荐顺序

重试通常放在限流与熔断之后：

```text
middlewares = ["auth-jwt", "rate-limit", "circuit-breaker", "retry"]
```
